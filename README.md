## Overview

The Knowledge Tier Framework provides a structured approach to classifying projects based on their information sensitivity requirements. It integrates with [Epistemic Guardrails](https://github.com/theios-research-institute/epistemic-guardrails-for-ai-agents) to enforce appropriate boundaries for AI agent access.

---

## The Four Tiers

| Tier | Name | Memory | Agents | Cloud Sync | Use Cases |
|------|------|--------|--------|------------|-----------|
| **1** | **Restricted** | OFF | None | No | Trade secrets, pre-patent, proprietary research |
| **2** | **Confidential** | Enabled | Limited | Limited | Competitive advantage, internal R&D |
| **3** | **Internal** | Enabled | Project-specific | Local only | Client work, business operations |
| **4** | **Public** | Enabled | All | Yes | Open source, documentation, tutorials |

---

## Tier Definitions

### Tier 1: Restricted

**For:** Information requiring maximum protection

- Trade secrets and proprietary algorithms
- Pre-patent research and inventions
- Breakthrough discoveries before publication
- Legal and personal sensitive data

**Requirements:**
- Memory MUST be disabled
- No external agent access
- No cloud synchronization
- Manual initialization only (no automated scaffolding)
- Complete isolation from other projects
- Declare allowed push destinations with `ALLOWED_REMOTES` (see [Outbound Action Protection](#outbound-action-protection))

**Legal basis:** Trade secret law requires "reasonable measures to maintain secrecy." AI memory retention may constitute inadequate protection.

---

### Tier 2: Confidential

**For:** Information providing competitive advantage

- Internal R&D projects
- Business strategy documents
- Proprietary methodologies
- Pre-release product features

**Requirements:**
- Memory enabled for continuity
- Limited agent access (documentation, formatting)
- Algorithm details reference local files only
- Session handoffs can sync to private cloud

---

### Tier 3: Internal

**For:** Business operations and client work

- Client projects with confidentiality agreements
- Internal tools and utilities
- Business processes
- Team collaboration documents

**Requirements:**
- Memory enabled
- Project-specific agents allowed
- No cross-project references to client data
- Local-only session handoffs

---

### Tier 4: Public

**For:** Information intended for broad access

- Open source projects
- Public documentation
- Tutorials and guides
- Marketing materials

**Requirements:**
- Full memory access
- All agents available
- Full cloud sync permitted
- Can be referenced from other projects

## Project Structure

Each tier template includes:

```
project-root/
├── .epistemic-tier               # Tier configuration (all tiers)
├── .claude_project_config.json   # Project metadata (Tier 1)
├── SECURITY_PROTOCOL.md          # Security guidelines (Tier 1 only)
└── init.sh                       # Environment verification (Tier 1)
```

**Note:** Tier 1 (Restricted) includes additional security files. Tiers 2-4 include only the `.epistemic-tier` configuration file.

## Usage

### Initialize a New Project

```bash
# Interactive tier selection
init-project-tier

# Or specify tier directly
init-project-tier --tier 2

# Initialize in specific directory
init-project-tier --tier 3 ~/projects/client-work
```

### Per-Project Configuration

Create a `.epistemic-tier` file in your project root:

```bash
# For Tier 1 (Restricted)
TIER=restricted
MEMORY_REQUIRED=off
AGENTS_ALLOWED=none
CLOUD_SYNC=no
ALLOWED_REMOTES=github.com/my-org/my-repo

# For Tier 2 (Confidential)
TIER=confidential
MEMORY_REQUIRED=on
AGENTS_ALLOWED=limited
CLOUD_SYNC=limited
#ALLOWED_REMOTES=

# For Tier 3 (Internal)
TIER=internal
MEMORY_REQUIRED=on
AGENTS_ALLOWED=project
CLOUD_SYNC=local
#ALLOWED_REMOTES=

# For Tier 4 (Public)
TIER=public
MEMORY_REQUIRED=on
AGENTS_ALLOWED=all
CLOUD_SYNC=yes
#ALLOWED_REMOTES=
```

### Check Project Tier

```bash
# In any project directory
cat .epistemic-tier
```

## Outbound Action Protection

The `ALLOWED_REMOTES` field in `.epistemic-tier` restricts where code can be pushed, published, or deployed. This prevents accidental pushes to wrong organizations, registries, or servers.

### Configuration

Add `ALLOWED_REMOTES` to your `.epistemic-tier` file with a comma-separated list of allowed destination patterns:

```bash
# Exact repository (works with any git host)
ALLOWED_REMOTES=github.com/my-org/my-repo
ALLOWED_REMOTES=gitlab.com/my-group/my-repo
ALLOWED_REMOTES=bitbucket.org/my-team/my-repo

# Multiple destinations (comma-separated)
ALLOWED_REMOTES=github.com/my-org/my-repo,gitlab.com/my-group/my-repo

# Wildcard — any repo under an org/group
ALLOWED_REMOTES=github.com/my-org/*
ALLOWED_REMOTES=gitlab.com/my-group/*

# Self-hosted servers
ALLOWED_REMOTES=git.internal.company.com/*
ALLOWED_REMOTES=192.168.1.50:*

# Package registries and S3 buckets
ALLOWED_REMOTES=registry.npmjs.org
ALLOWED_REMOTES=s3://my-deploy-bucket
```

### Behavior

| `ALLOWED_REMOTES` Value | Effect |
|-------------------------|--------|
| Not present | No outbound checking (backwards compatible) |
| Empty (`ALLOWED_REMOTES=`) | No outbound checking (backwards compatible) |
| Populated | Outbound actions checked against list; blocked if no match |

### Pattern Matching

- **Exact substring**: `github.com/my-org/my-repo` matches any URL containing that string
- **Trailing wildcard**: `github.com/my-org/*` matches any URL containing `github.com/my-org/`
- **All URL formats handled automatically**: `git@github.com:my-org/my-repo.git` is normalized to `github.com/my-org/my-repo` before matching, so one pattern covers HTTPS, SSH (`git@host:path`, `ssh://`), and `git://` protocol remotes

### What Gets Checked

When paired with [Epistemic Guardrails](https://github.com/theios-research-institute/epistemic-guardrails-for-ai-agents), the following commands are intercepted: `git push`, `git remote add`, `git remote set-url`, `gh repo create`, `npm publish`, `cargo publish`, `pip/twine upload`, `rsync`, `scp`, `aws s3 cp/sync`.

## Integration with Epistemic Guardrails

This framework works seamlessly with [Epistemic Guardrails](https://github.com/theios-research-institute/epistemic-guardrails-for-ai-agents):

1. **Epistemic Guardrails** enforces memory boundaries based on tier
2. **Knowledge Tier Framework** provides the classification structure and templates
3. Together they form a complete epistemic governance system for AI agents

Install both for full protection:

```bash

# Install Knowledge Tier Framework (provides structure)
```

## Best Practices

### Tier Selection Guide

Ask these questions to determine the appropriate tier:

1. **Would disclosure cause significant harm?** → Tier 1 (Restricted)
2. **Does it provide competitive advantage?** → Tier 2 (Confidential)
3. **Is it client or internal business data?** → Tier 3 (Internal)
4. **Is it intended for public consumption?** → Tier 4 (Public)

### Upgrading Tiers

You can always move to a MORE restrictive tier:
- Tier 4 → Tier 3 → Tier 2 → Tier 1 ✅

Moving to a LESS restrictive tier requires review:
- Tier 1 → Tier 2 ⚠️ Review for residual sensitive data

### Cross-Tier References

- Tier 1 projects: NO references to or from other projects
- Tier 2 projects: Reference Tier 3-4 only (read-only)
- Tier 3 projects: Reference Tier 4 only
- Tier 4 projects: Can be referenced by all tiers

## Troubleshooting

### "Command not found" after installation

```bash
# Reload your shell configuration
source ~/.zshrc  # or ~/.bashrc
```

### Templates not found

1. Verify templates installed: `ls ~/.claude/templates/`
2. Re-run installer: `./install.sh`

### init.sh fails verification

1. Check the specific error message
2. For Tier 1: Ensure memory is OFF (`memory-status`)
3. Verify `.epistemic-tier` file exists and has correct values

### Wrong tier detected

1. Check for `.epistemic-tier` in current and parent directories
2. The closest `.epistemic-tier` to your working directory takes precedence
3. Remove or update conflicting tier files

### Permission denied errors

```bash
# Make scripts executable
chmod +x ~/.claude/scripts/*.sh
chmod +x ~/.claude/templates/*/init.sh
```
